### US-002: Keep send_embed payload compatibility and add username guard
Status: in_progress
Depends on: US-001

Description:
As an integrator, I want send_embed to remain compatible with existing caller payloads so current site requests continue to work without regressions.

Acceptance Criteria:
- [ ] POST /send_embed continues accepting payload with field username only
- [ ] Input normalization for username is added: empty string, null, undefined, or whitespace resolves to fallback 'Игрок'
- [ ] Example: payload { "username": "Player123" } posts message successfully with player name 'Player123'
- [ ] Negative case: payload { "username": "   " } does not crash and posts message with name 'Игрок'
- [ ] Route returns non-error response for compatible old payload shape used by site backend
